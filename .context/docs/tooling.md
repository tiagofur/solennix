# Tooling & Productivity Guide

This guide outlines the scripts, automation, and editor configurations designed to keep developers efficient while working on the Solennix codebase. By standardizing our environment and leveraging shared automation, we ensure consistent code quality and reduce friction during development.

## Required Tooling

To develop, build, and maintain the application, the following tools are required:

*   **Node.js (v18.x or higher):** The runtime environment for our development server and build pipeline.
    *   *Installation:* Use [nvm](https://github.com/nvm-sh/nvm) to manage versions. `nvm install 18 && nvm use 18`.
*   **npm (v9.x or higher):** Package manager for dependency management.
*   **Vite:** Powers our local development server and optimized production builds. It provides Hot Module Replacement (HMR) for instant feedback.
*   **Supabase CLI:** Used for managing database migrations, generating TypeScript types, and local development.
    *   *Installation:* `npm install -g supabase`
    *   *Purpose:* Synchronizing the `src/types/supabase.ts` file with the remote database schema.
*   **TypeScript:** Provides static type checking across the application.
    *   *Version:* See `package.json` for the currently pinned version.

## Recommended Automation

### Development Scripts
The following commands are available via `npm run`:

*   `dev`: Starts the Vite development server with HMR. Use this for all active coding.
*   `build`: Runs the TypeScript compiler (`tsc`) and then the Vite build process to generate a production-ready `dist` folder.
*   `lint`: Executes ESLint to find and fix code style issues and potential bugs.
*   `preview`: Starts a local server to preview the production build generated by the `build` script.

### Type Generation
To keep the application's database types in sync with the Supabase backend, run the following command when the schema changes:

```bash
# Replace <project-id> with your Supabase project ID
npx supabase gen types typescript --project-id <project-id> > src/types/supabase.ts
```
This ensures that the `Database` interface in `src/types/supabase.ts` remains accurate, preventing runtime errors in services like `eventService.ts` or `inventoryService.ts`.

### Formatting
The project uses **ESLint** for code quality. It is recommended to run the linter before pushing code:
```bash
npm run lint
```

## IDE / Editor Setup

We recommend using **Visual Studio Code (VS Code)** for the best development experience.

### Recommended Extensions
*   **ESLint:** Integrates linting directly into the editor. Set `editor.codeActionsOnSave` to `source.fixAll.eslint` in your settings.
*   **Tailwind CSS IntelliSense:** Provides autocomplete and linting for Tailwind classes used in our components (e.g., `src/components/Layout.tsx`).
*   **PostCSS Language Support:** Essential for processing the global CSS files.
*   **Prettier - Code Formatter:** Although ESLint is used for linting, Prettier is recommended for consistent indentation and formatting.

### Workspace Settings
Create or update `.vscode/settings.json` in the root directory:
```json
{
  "editor.formatOnSave": true,
  "editor.defaultFormatter": "esbenp.prettier-vscode",
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  },
  "tailwindCSS.includeLanguages": {
    "typescript": "javascript",
    "typescriptreact": "javascript"
  }
}
```

## Productivity Tips

### Supabase Development
*   **Local Emulation:** For complex database work, consider using the Supabase local development environment (`supabase start`) to test migrations before applying them to production.
*   **Auth Checks:** Use the `isSupabaseConfigured()` utility from `src/lib/supabase.ts` to quickly debug connection issues in new environments.

### Terminal Aliases
If you frequently work on this project, adding these aliases to your `.zshrc` or `.bashrc` can save time:
```bash
alias ev-dev="npm run dev"
alias ev-build="npm run build"
alias ev-lint="npm run lint"
```

### PDF Debugging
When working on `src/lib/pdfGenerator.ts`, use the `EventSummary.tsx` "Print" functionality to test layout changes. Since PDF generation happens client-side, you can inspect the generated objects in the browser console before they are passed to the PDF engine.
